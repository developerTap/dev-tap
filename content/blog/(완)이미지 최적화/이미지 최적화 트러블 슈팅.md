# 개요

  

기존 홈페이지에서 페이지 로드 시 높은 화소 이미지로 인한 느린 랜딩 속도로 인해 UX 개선이 절실히 필요하였다. 차선책으로 이미지 압축을 통해 약간의 개선을 하였지만, 그래도 높은 용량의 이미지 처리와 떨어진 퀄리티로 인해 디자이너, 개발자의 만족도가 낮은 상태였고, 퀄리티를 개선하는 방향을 찾다 Next.js(v12.2)의 이미지 최적화 서버를 활용 가능하다고 판단하여 적용하게 되었다.

  

사실 Next.js의 `Next/Image` 컴포넌트는 이미 사용 중이었으나, 다양한 이유로 인해 운영환경에서 최적화가 된 채로 배포가 되지 않고 있었다. 다른 사람들은 그냥 컴포넌트만 쓰면 알아서 optimize가 된다는데…. 왜 우리만…!? 저 버전의 CI/CD 환경에서 Next.js의 이미지 최적화 성공 과정 사례를 적어보려한다.

  

# Trouble Shooting

  

## `next/future/image` 컴포넌트 버전 이슈

  

Next.js는 자체적으로 이미지 최적화를 위해 `Next/Image` 컴포넌트를 제공한다. 하지만 `v12.2`에선 강제로 `span` 태그가 감싸져, 불피요한 css 제어가 필요했지만, 다행히 `v12.2` 버전에서 `v13`에서 제공되는 [`next/future/image`](https://nextjs.org/blog/next-12-2#improvements-to-nextimage)태그가 지원 된다는 것을 보고 바로 사용하게 되었다. 그런데 여기서 두가지 문제가 발생하게 되었다.

  

1. Next 이미지 컴포넌트 사용 이슈

- 디자이너 쪽에서 제공되는 이미지를 넣었더니 과한 최적화가 발생되어 쓸수 없었던 점.(next 이미지 컴포넌트에선 `width, height`에 따라 자동으로 이미지 배율(1x, 2x, 3x...) 조절이 된다)

- 퀄리티를 살리기위해 `unoptimized`를 활성화 시켜 원본 이미지를 불러오니 끔찍한 이미지 렌더링 속도를 경험하게됨.

- 이미지를 압축하여 사용했지만, `optimized`보다 심각한 저화질의 이미지를 사용해야 했음.

  

2. 골치아프게 했던 `sharp` 경고문

> Warning: For production Image Optimization with Next.js, the optional 'sharp' package is strongly recommended. Run 'npm i sharp', and Next.js will use it automatically for Image Optimization.

  

next에선 이미지 최적화시 [squoosh](https://squoosh.app/)와 [sharp](https://sharp.pixelplumbing.com/)라는 이미지 최적화 라이브러리가 내장되어있는데, 개발 런타임에선 squoosh가 사용되고, 빌드 후 런타임 환경에선 sharp라이브러리로 대체된다. 관련해선 올리브영에서[NEXT.JS의 이미지 최적화는 어떻게 동작하는가?](https://oliveyoung.tech/blog/2023-06-09/nextjs-image-optimization/)에서 설명이 잘 되어있어서 참고해보면 좋을 것 같다.

  

어쟀든 더 확실한 성능 차이(약 6배 차이)가 있기 때문에 sharp를 필수 적으로 사용하게 된다. 이 프로젝트는 yarn-berry의 zero-install을 사용하고 있기 때문에, `.yarn`에 들어가지지 않은 sharp 모듈은 패키지가 "분리"되어 `unplugged` 폴더로 주입되어 패키지를 install하게 된다. 사실 개발환경에선 이것에 대해 크게 상관하지도 않았고(dev 모드에선 next는 squoosh를 사용하기 때문), 일반적인 상황에선 `unplugged` 없어도 무방하거니와 폴더 자체를 git에 올리지 않기에 대수롭지 않게 생각했다. 하지만 여기서 다음 문제가 발생하게 된다.

  

## standalone과 sharp 이슈

  

프로젝트에서 프로덕션 배포에 "필요한" 파일만 자동으로 복사되는 [`standalone`](https://nextjs.org/docs/pages/api-reference/next-config-js/output)을 사용하고 있다. 그런데 이 standalone이 발목을 잡게 되었는데 빌드 후 standalone의 `.yarn/cache`에서 sharp가 보이지 않는다...? 개발환경에서만 `unplugged`로 빠진다고 생각했는데 그게 아니었던 것이다.

  

![Alt text](.standalone.png)

  

확인해보니 아래와 같은 내용을 [공식 문서](https://nextjs.org/docs/messages/sharp-missing-in-production)에서 안내가 되고 있었다.

  

> If sharp is already installed but can't be resolved, set the NEXT_SHARP_PATH environment variable such as export NEXT_SHARP_PATH=/tmp/node_modules/sharp. Then, build your project with next build. Finally, restart the server with either next start for production mode or node .next/standalone/server.js for standalone mode.

  

> Note: This is not necessary for Vercel deployments, since sharp is installed automatically for you.

  

찾아보니 `NEXT_SHARP_PATH`를 지정해서 직접 패키지 모듈에 들어있는 sharp 패키지를 직접 접근하라는 뜻이었다. (그렇지만 Vercel 배포를 사용한다면 알아서 됩니다 ^^ 이 Note가 꽤나 얄미웠다)

  
  

당시에는 `node_modules`에 대한 솔루션과 github issue에서도 `.yarn`를 사용하는 유저들을 위한 답변을 찾을 수가 없었다. sharp 모듈을 지정하고 싶어도 그럴수가 없는 상황이었는데, zero-install을 하면서 docker에 올릴때도 기존 `.yarn`도 같이 올리게 되는데 그안에 있던 `unppluged`내의 sharp 패키지를 정적 경로로 가져오게 해봤다.

  

![Alt text](sharp.png)

  

다행인지 모르겠지만 아래와 같이 정적 경로로 지정하니 sharp라이브러리를 가져와 최적화 기능이 정상 작동했다. 하지만 이게 정상적인 방법이 아니라는 건 온몸이 알고 있기에 이 상태로 1차 배포 후 다시 확인해보았다.

  

```DockerFile

ENV NEXT_SHARP_PATH=/app/.yarn/unplugged/sharp-npm-0.32.6-6b3822c437/node_modules/sharp

```
